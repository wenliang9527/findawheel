## 🚀 批次 3.1：可靠性基座 + 数据源扩展

**目标：** 建立缓存/重试/去重基础设施，同时扩展 GitLab/PyPI/Libraries.io 三个数据源。

### 设计决策

#### A. 本地缓存层

| 决策点 | 方案 | 理由 |
|:-----|:-----|:-----|
| 存储位置 | `~/.findawheel/cache/`（用户目录） | stdio 每次启动是新进程，必须磁盘持久化才能跨会话复用 |
| 存储格式 | JSON 文件，按 cache key 命名 | 简单可调试，无需引入 SQLite |
| cache key | `sha1(query + intent + ecosystem + limit)` | 唯一标识一次查询 |
| TTL | 默认 1 小时，`FINDAWHEEL_CACHE_TTL_MS` 可配 | 平衡新鲜度和性能 |
| 默认开关 | 默认开启，`FINDAWHEEL_CACHE_ENABLED=false` 可关 | 默认受益，可逃生 |
| 失效策略 | 纯 TTL，不支持手动刷新 | YAGNI，重启进程即可清缓存 |
| 缓存层级 | 缓存 `Wheel[]` 最终结果（归一化+排序后） | 命中时跳过全部后续处理，最快 |

#### B. 重试策略

| 决策点 | 方案 | 理由 |
|:-----|:-----|:-----|
| 触发条件 | 网络错误（fetch reject）+ 5xx | 瞬时错误才值得重试 |
| 不重试 | 4xx（含 401/402/403/404/429） | 4xx 重试无意义；429 重试会雪崩 |
| 策略 | 指数退避，base 300ms，最多 2 次 | 总耗时增加 < 1s，可接受 |
| 实现位置 | `util/http.ts` 的 `httpGet` 内部 | 所有 adapter 自动受益 |
| MCP fetch | `webSourceAdapter` 用原生 fetch，单独包一层 retry | Exa/Tavily 也受益 |

#### C. 请求去重（in-flight dedup）

| 决策点 | 方案 | 理由 |
|:-----|:-----|:-----|
| 范围 | 同一进程内，同一 cache key 的并发请求 | stdio 单进程，进程内去重足够 |
| 实现 | `Map<key, Promise<Wheel[]>>`，命中复用 Promise | 简单 |
| 缓存命中 | 直接返回，不触发去重 | 缓存更快 |
| 实现 | 合并到 `cache.ts`，避免文件碎片 | 单一职责：缓存+去重都是"请求合并" |

#### D. GitLab 源

| 决策点 | 方案 |
|:-----|:-----|
| API | `GET https://gitlab.com/api/v4/search?scope=projects&search=<q>&order_by=stars` |
| 鉴权 | 无需 token（公开搜索）；可选 `GITLAB_TOKEN` 提升限流 |
| 语法 | 不支持 NOT/引号（类似 Gitee） |
| 字段映射 | `path_with_namespace`→name, `web_url`→url, `star_count`→stars, `last_activity_at`→lastUpdated |

#### E. PyPI 源

| 决策点 | 方案 |
|:-----|:-----|
| API | HTML 解析 `https://pypi.org/search/?q=<q>` |
| 容错 | 解析失败返回空数组，不抛错（HTML 结构可能变更） |
| 字段 | 包名、描述、链接；无 stars/downloads |
| 风险 | HTML 结构变更需监控；解析用正则或简易 DOM 提取 |

#### F. Libraries.io 源

| 决策点 | 方案 |
|:-----|:-----|
| API | `GET https://libraries.io/api/search?q=<q>&api_key=<key>` |
| 鉴权 | 需要 `LIBRARIES_IO_API_KEY` |
| 价值 | 一次查询覆盖 30+ 包管理器（npm/pypi/rubygems/cargo/maven...） |
| 字段映射 | `name`→name, `description`→description, `homepage`→url, `stars`→stars, `language`→language |

### 任务清单（实现时细化为 TDD 步骤）

> 以下为任务分解，每项在进入实现时按 writing-plans 规范展开为 TDD 步骤（写失败测试 → 实现 → 通过 → 提交）。

#### Task 1: 缓存 + 去重层
- `src/cache/cache.ts`：`get(key)` / `set(key, val)` / `dedupe(key, fn)`
- 磁盘读写（`~/.findawheel/cache/<key>.json`），TTL 检查
- in-flight Map 去重
- 测试：TTL 过期、命中、去重并发只发一次

#### Task 2: 重试包装器
- `src/util/retry.ts`：`withRetry(fn, { retries: 2, baseMs: 300 })`
- 仅对网络错误 + 5xx 重试；4xx 直接抛
- 测试：5xx 重试成功、4xx 不重试、达到上限抛错

#### Task 3: http.ts 集成重试
- `httpGet` 内部用 `withRetry` 包装 fetch
- `webSourceAdapter` 的 Exa/Tavily fetch 也包 `withRetry`
- 测试：mock fetch 验证重试行为

#### Task 4: findWheelTool 集成缓存+去重
- 计算 cache key → 先查缓存 → 未命中则去重执行 → 写缓存
- `FINDAWHEEL_CACHE_ENABLED=false` 时跳过
- 测试：缓存命中跳过 adapter、未命中写缓存

#### Task 5: env.ts 扩展配置
- 新增 `cacheEnabled`/`cacheTtlMs`/`gitlabToken`/`librariesIoApiKey`
- 测试：默认值、env 覆盖

#### Task 6: GitLab adapter
- `GitlabRawResult` 类型 + adapter + Normalizer 分支
- SearchOpts 加 `gitlabToken`
- 测试：搜索语法、字段映射、403 限流处理

#### Task 7: PyPI adapter
- `PypiRawResult` 类型 + adapter（HTML 解析）+ Normalizer 分支
- 测试：HTML 解析、解析失败容错返回空

#### Task 8: Libraries.io adapter
- `LibrariesIoRawResult` 类型 + adapter + Normalizer 分支
- SearchOpts 加 `librariesIoApiKey`
- 测试：API 调用、字段映射、无 key 返回空

#### Task 9: server.ts 注册 + .env.example + 文档
- 注册 3 个新 adapter
- `.env.example` 加 4 个配置项
- 更新 README 数据源表/环境变量表/路线图
- 端到端冒烟测试

### 风险与缓解

| 风险 | 缓解 |
|:-----|:-----|
| PyPI HTML 结构变更导致解析失效 | 解析失败返回空数组，不阻断；日志告警 |
| 缓存过期数据误导用户 | TTL 默认 1h，且 `FINDAWHEEL_CACHE_ENABLED=false` 可逃生 |
| 新增 3 源增加总体延迟 | 用 `Promise.allSettled` 并行，单源超时 `timeoutMs` 不变 |
| Libraries.io 需要 API key | 零配置兼容：未配置时跳过该源 |
